# Self-hosted mail with Mailcow

How we run our own mail for `tzhk.dev` (and client domains): real mailboxes +
webmail **and** an SMTP server this CMS sends through (password resets, invites).

The CMS side is already wired: when `SMTP_HOST` is set, Payload sends through it
(`@payloadcms/email-nodemailer`, see `src/payload.config.ts`). This doc covers
the server that `SMTP_HOST` points at.

## TL;DR — the decisions

| Topic | Decision | Why |
|-------|----------|-----|
| Software | **Mailcow** (mailcow-dockerized) | Full suite (Postfix + Dovecot + SOGo webmail + Rspamd + DKIM) with a real **web admin UI**, multi-domain, easy to add client domains |
| Where it runs | **A dedicated second Hetzner VPS** — NOT the Coolify node | Mailcow wants ports 80/443 which Coolify's proxy already owns; mail also wants a clean host + its own PTR/reputation |
| Sizing | CX22 / CPX21 (≈3 vCPU, 4 GB) minimum; 6 GB+ ideal | Mailcow is heavy; ClamAV can be disabled to fit 4 GB |
| Hostname | `mail.tzhk.dev` (changeable later) | One A record + a fresh cert to move it |
| Admin model | We administer everything ourselves | Create client domains/mailboxes in the Mailcow UI, hand clients only their webmail login |

> **Do not run Mailcow on the Coolify node.** Coolify's Traefik holds 80/443
> (webmail, admin UI, ACME), so it collides head-on with Mailcow. Mixing mail
> with web apps also means a Coolify rebuild/outage takes down every client's
> mail. Keep mail isolated on its own box.

## Architecture

```
 VPS #1  (existing)                         VPS #2  (new, mail only)
 ┌──────────────────────┐   SMTP 587/STARTTLS  ┌────────────────────────────┐
 │ Coolify              │ ───────────────────► │ Mailcow  (mail.tzhk.dev)   │
 │  - tzhk-cms (Payload)│                      │  Postfix  send/receive     │
 │  - other sites       │                      │  Dovecot  mailboxes/IMAP   │
 └──────────────────────┘                      │  SOGo     webmail          │
                                               │  Rspamd   antispam + DKIM  │
        clients ─── IMAP 993 / webmail 443 ──► └────────────────────────────┘
```

## 1. Provision the mail VPS (Hetzner)

1. Create a new server (CX22 or CPX21, Ubuntu 24.04 LTS). Give it a **dedicated
   IP**.
2. **Reverse DNS (PTR):** in Hetzner Cloud Console → the server → *Networking* →
   set rDNS for the IPv4 (and IPv6) to `mail.tzhk.dev`. Without a matching PTR,
   Gmail/Outlook will reject or spam-folder your mail.
3. Open the firewall for: `25, 80, 110, 143, 443, 465, 587, 993, 995, 4190`
   (SMTP/SMTPS/submission, IMAP(S), POP3(S), webmail/admin, ManageSieve). Hetzner
   unblocks port 25 on request — confirm it's open before going live.
4. Set the system hostname:
   ```bash
   hostnamectl set-hostname mail.tzhk.dev
   ```

## 2. DNS records (set on `tzhk.dev`)

Replace `MAIL_IPV4` / `MAIL_IPV6` with the new server's IPs.

| Type | Name | Value | Notes |
|------|------|-------|-------|
| A | `mail` | `MAIL_IPV4` | the mailserver host |
| AAAA | `mail` | `MAIL_IPV6` | if you use IPv6 (then PTR for v6 too) |
| MX | `@` | `mail.tzhk.dev.` (priority `10`) | routes inbound mail |
| TXT (SPF) | `@` | `v=spf1 mx ~all` | authorizes the MX to send |
| TXT (DMARC) | `_dmarc` | `v=DMARC1; p=quarantine; rua=mailto:postmaster@tzhk.dev` | start at `quarantine`, tighten to `p=reject` later |
| TXT (DKIM) | `dkim._domainkey` | *(generated by Mailcow)* | copy the exact value from the Mailcow UI after step 3 |
| CNAME | `autodiscover` | `mail.tzhk.dev.` | Outlook autoconfig |
| CNAME | `autoconfig` | `mail.tzhk.dev.` | Thunderbird autoconfig |

> SPF/DMARC must exist **before** you start sending in volume. DKIM is generated
> by Mailcow (next step), then pasted back into DNS.

## 3. Install Mailcow

Official docs: https://docs.mailcow.email/getting-started/install/

```bash
# Docker + compose plugin first (Mailcow ships its own stack)
curl -sSL https://get.docker.com/ | sh

cd /opt
git clone https://github.com/mailcow/mailcow-dockerized
cd mailcow-dockerized
./generate_config.sh        # asks for MAILCOW_HOSTNAME -> mail.tzhk.dev, timezone
```

Optional, to fit 4 GB RAM — disable ClamAV (and optionally Solr) in
`mailcow.conf`:

```
SKIP_CLAMD=y
SKIP_SOLR=y
```

Then bring it up:

```bash
docker compose pull
docker compose up -d
```

Mailcow gets its TLS cert automatically via Let's Encrypt for `mail.tzhk.dev`
(needs the A record + ports 80/443 reachable).

Admin UI: `https://mail.tzhk.dev` — default login `admin` / `moohoo`.
**Change this password immediately.**

## 4. Add the domain, mailboxes, and DKIM

In the Mailcow admin UI:

1. **Configuration → Mail → Domains → Add domain:** `tzhk.dev`.
2. **Configuration → Mail → Mailboxes → Add mailbox:**
   - `no-reply@tzhk.dev` — used by this CMS to send (give it a strong password).
   - any human mailboxes you want (e.g. `hello@tzhk.dev`).
3. **Configuration → ARC/DKIM keys:** generate a key for `tzhk.dev` (selector
   `dkim`), then copy the public key and create the
   `dkim._domainkey.tzhk.dev` TXT record from step 2.
4. **Check it:** Mailcow's *Domain → DNS check* page shows green/red for every
   record (MX, SPF, DKIM, DMARC, PTR). Get them all green before sending real
   mail.

## 5. Point the CMS at it

In Coolify, on the `tzhk-cms` service, set these env vars (the code already
reads them — `src/payload.config.ts`):

```
SMTP_HOST=mail.tzhk.dev
SMTP_PORT=587
SMTP_SECURE=            # leave blank — 587 uses STARTTLS, not implicit TLS
SMTP_USER=no-reply@tzhk.dev
SMTP_PASS=<the mailbox password from step 4>
EMAIL_FROM=no-reply@tzhk.dev
EMAIL_FROM_NAME=TZHK
```

Redeploy. Trigger a password-reset email from the admin to confirm delivery.
(With `SMTP_HOST` empty, Payload just logs emails to the console — handy for
local dev.)

## 6. Adding a client (we admin it)

For a client who has no mail yet:

1. Mailcow → **Add domain** `clientdomain.com`.
2. On **the client's domain's DNS**, add the same record set as step 2 (MX →
   `mail.tzhk.dev`, SPF, DMARC, and the DKIM record Mailcow generates for that
   domain).
3. Mailcow → **Add mailbox** `name@clientdomain.com`.
4. Hand the client only their webmail URL (`https://mail.tzhk.dev/SOGo`) +
   credentials. They read/send from the browser; everything else stays with us.

A client domain can also be the `EMAIL_FROM` for a tenant if we later want the
CMS to send "from" their domain — same SMTP server, just a mailbox on that
domain.

## Deliverability checklist

- [ ] PTR (rDNS) for the IP = `mail.tzhk.dev`
- [ ] MX → `mail.tzhk.dev`
- [ ] SPF `v=spf1 mx ~all`
- [ ] DKIM record matches the key in Mailcow
- [ ] DMARC present (`p=quarantine`, later `p=reject`)
- [ ] Mailcow *DNS check* all green
- [ ] Test against https://www.mail-tester.com (aim for 10/10)

## Maintenance

- Update Mailcow: `cd /opt/mailcow-dockerized && ./update.sh`
- Backups: `./helper-scripts/backup_and_restore.sh backup all` (cron it; ship
  off-box).
- Changing the hostname later: update the `mail` A record + PTR, set the new
  `MAILCOW_HOSTNAME` in `mailcow.conf`, re-run the cert flow, and update
  `SMTP_HOST` in Coolify.
